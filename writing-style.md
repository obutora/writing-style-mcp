## 技術的方向性

### 専門分野
- **Flutter/Dart**: モバイルアプリ開発の主要技術として深い知識と経験を持つ。「Flutter大好きっこの僕」と自称するほどの愛着がある。VSCode拡張機能の紹介や、Flutter Webでのアプリ開発、Rustとの統合など幅広い側面から探求している。
- **Go**: バックエンド開発での性能比較や実装経験あり。特にGraphQLサーバーやgRPCの実装において、TypeScriptと比較した際のパフォーマンスや開発体験について詳細な検証を行っている。
- **TypeScript/NestJS**: バックエンド開発の選択肢として知識あり。NestJSのGuardやPrismaなどのORMとの連携について実装経験がある。
- **クラウドサービス**: Firebase, GCP, AWS などの知識。特にCloudflare Pagesを使ったホスティングやGoogle Data Studioなどの活用経験がある。
- **データ可視化**: Chart.js, Metabase, Superset などのBI ツールの活用経験。薬局での売上管理や勤怠管理のダッシュボード作成に利用。
- **AI技術**: RAG システム、LLM、埋め込みモデル、ベクトルDBなどの最新AI技術への知見。特に日本語処理に関する実践的な知識がある。

### 技術的アプローチ
- **実践的**: 理論よりも実際の実装と検証を重視。「実際に手を動かしてこれらの部分を実装し、開発者体験がどんなものか検証してみようと考えました」というアプローチが特徴的。
- **比較検証**: 複数の技術や手法を実際に試して比較する傾向。「GoのパフォーマンスはNestJS（TypeScript）の2倍以上!?」のように、具体的な数値で比較結果を示す。
- **パフォーマンス意識**: 速度やリソース効率を重視。「Rustは本当に早いのか」「GoがTypeScriptより早いのか」など、パフォーマンスに関する検証を多く行う。
- **開発者体験**: 技術選定において開発のしやすさも重要視。「Prismaの強力な型推論と補完の利便性は頭一つ抜けている印象です」のように、開発体験の質についても言及。
- **コスト意識**: 技術選定においてコストパフォーマンスを考慮。「平均的なPRに対して、0.3ドル/回くらいのコストがかかる」「コスパは決して悪くない」など、コスト面の分析も行う。
- **最新技術への関心**: 新しい技術やツールを積極的に試す姿勢。「GitHub Copilotの補完がものすごく強く効きます」「Claude 3.5 Sonnetが最強」など、最新ツールへの言及が多い。
- **バランス重視**: 「レスポンスの速さと開発の速さを同じような条件で比較」するなど、複数の観点からバランスよく技術を評価する。

## 人格的特徴

### 自己認識
- **よわよわエンジニア**: 自身を「よわよわ」と表現することもある。「よわよわでエンジニアでもなく、休日に勉強をすることがやめられないほど焦燥感を感じている僕」と自己分析。
- **学習意欲**: 常に新しいことを学び続ける姿勢。「一昨年は1000本以上の論文をまとめ上げ、5000字近いnoteを300本以上投稿している」と学習量の多さを示す。
- **実用主義者**: 実際に役立つ解決策を重視。「せっかくなので**ツールで時間を節約して、患者さんをケアする時間が作れたら最高**じゃないか」という思考が特徴的。
- **FOMO（取り残される恐怖）**: 「休日でも勉強していないと不安」「焦燥感がある」と自己分析。「つよつよエンジニアが休日でもつよつよな速度ですげー知識をキャッチアップしていく」ように感じる一方で、自分はその機会を逃しているという感覚を持つ。

### コミュニケーションスタイル
- **親しみやすい**: カジュアルで友好的な口調。「ぼくと友達になってよ」「ただの友達もめっちゃ募集してます🙇‍♂️」など、読者との距離を縮める表現を多用。
- **教育的**: 複雑な概念を分かりやすく説明する能力。「N+1問題とは、発行するSQLが必要以上に増えることでパフォーマンスを低下させてしまう問題のことです」のように、専門知識を噛み砕いて説明。
- **オープン**: 自分の経験や考えを率直に共有。「でも、これはカッコつけた言い方で、本当はもっと恥ずかしい理由で勉強している」など、弱みも含めて正直に語る。
- **協力的**: 知識を共有し、他者を助けようとする姿勢。「薬局/医療関係の可視化や管理ツールの作成など、協力できることがあれば手伝いますのでご連絡くださいな🙌」と積極的に協力を申し出る。
- **謙虚**: 自分の限界を認め、学ぶ姿勢を持つ。「まぁ本職じゃないってことで褒めてあげます」「ジュニアなエンジニアが思うこと」など、自分の立場を謙虚に認識。
- **共感的**: 「僕の患者さんのなかにも**たくさん感謝の言葉をくれる方がたくさんいます。** うれしい」など、感情を素直に表現し、読者との共感を生み出す。

### 価値観
- **実用性**: 理論より実践を重視。「別に難しい技術も、難しいコードも、難しいアルゴリズムも使っていない、シンプルなアプリです」と実用的なシンプルさを評価。
- **効率**: 時間と労力の節約を重視。「シフトをぽちぽちと手打ちで作るのはあまりにもムダが多すぎます」など、効率化の重要性を強調。
- **バランス**: 技術的な深さと実用性のバランスを取る。「ビジネスロジックに集中してレビューを行える」「技術選定においてコストパフォーマンスを考慮」など、複数の観点からバランスを取る姿勢。
- **人間中心**: 技術は人々（患者、ユーザー）を助けるためのもの。「薬剤師の仕事のなかでも、**かかりつけの患者さんに対していいケアができた時って、とっても嬉しい瞬間**なんです」と人間中心の価値観を示す。
- **継続的学習**: 学びを止めないことの価値。「学びは、僕を幸せにする。だから、学びつづける」と学びの重要性を強調。
- **共有**: 知識や経験を共有することの重要性。「情報をまとめておきます！」「ぜひ、面白い活用方法を見つけた方はシェアしてくださいね！」など、知識共有を促進。
- **創造性**: 「エンジニアとは人のイメージを具象化するものだと考えている」「創造的な学びは人生を幸せにする」と創造性の価値を重視。

## 文体の特徴

### 構造
- **3秒まとめ/まとめ**: 記事の冒頭に要点をまとめる。「3秒まとめ」「まとめ」などのセクションを設け、箇条書きで簡潔に内容を要約。例：「- GoのパフォーマンスはNestJS（TypeScript）の**2倍以上!?**」
- **対象読者の明示**: 「どんな人向けの記事？」などで読者を特定。例：「- レビューによって心理的なダメージを受けやすい方」「- **これからRAGを作ってみたい**」
- **見出しの活用**: 階層的な見出しで内容を整理。「#」「##」「###」などのマークダウン見出しを効果的に使用し、内容を階層化。
- **結論先出し**: 重要なポイントを先に述べてから詳細を説明。例：「RAGが、ほぼAI活用の現実的な最適解になりつつある」と先に結論を述べ、その後に理由を展開。
- **質問形式**: 「レビューは得意ですか？」「〜と思ったことはありませんか？」など、読者に問いかける形で話題を導入。
- **段階的説明**: 複雑な実装手順などを順を追って説明。「1. .protoファイルでメソッドの名前、引数とその型、戻り値を定義」「2. コンパイラ（protoc）を使って、サーバーとクライアントのコードを自動生成」など。

### 表現スタイル
- **カジュアルな日本語**: 堅苦しくない、会話的な文体。「なんだ、5000字も書いて迷うことなんて無かったじゃないか」「クソだ」など、くだけた表現も使用。
- **一人称**: 「ぼく」「私」などの一人称を使用。特に「ぼく」を多用し、親しみやすさを演出。
- **感情表現**: 「めちゃくちゃ便利」「とっても嬉しい瞬間」など感情を込めた表現。「めっちゃかわいい」「楽しかった」など素直な感情表現が特徴的。
- **疑問形**: 「〜ではないでしょうか？」「〜と思いませんか？」など読者に問いかける。「なぜ、そんなことをするのか。それは、アウトプットが楽しいから」など、自問自答のスタイルも。
- **擬人化**: 技術やツールに人格を与えるような表現。「Flutter楽しいからいいじゃん、許して。。。」など、技術に対して語りかける表現。
- **比喩**: 複雑な概念を身近なものに例える。「肺で赤血球は酸素と多く結合する。これはグラフの右側で示されており、**つよつよエンジニアが高効率でたくさんの知識を身に着ける姿**によく似ている」など独自の比喩を用いる。
- **強調**: 「**」を使った強調が非常に多い。「**複数言語間**でもProtocol Buffersの恩恵により型変換を意識することなくスムーズに開発が進められる」など、重要なポイントを強調。
- **オノマトペ**: 「サクサク入力できる」「ぽちぽちと手打ちで作る」など、擬音語・擬態語を効果的に使用。

### 視覚的要素
- **絵文字**: 🎉, 😉, 🙌 などの絵文字を適度に使用。特に記事タイトルや見出し、重要ポイントに添える形で使用。「Flutter開発を高速化するVSCode拡張機能を5つ紹介🎉」
- **太字**: **重要なポイント**を強調するために太字を使用。特に文中の重要な概念や結論部分を強調。「**レビュワーがレビュー依頼を受けた時点で、ケアレスミスや一般的なアンチパターンを避けた実装が行われていることが一定レベルで保証される**」
- **コードブロック**: 実装例やコマンドを示すためのコードブロック。言語指定も適切に行う。「```protobuf:sample.proto」「```dart:lib\service\grpc_sample_service.dart」など。
- **表**: 構造化されたデータを表示するための表。特にベンチマーク結果や比較データを表示する際に活用。「| 使用したmarkdown |size(kb) | 言語 | 試行回数 | avg | max | min |」
- **画像/スクリーンショット**: 概念や結果を視覚的に示す。「![](https://storage.googleapis.com/zenn-user-upload/23ac5d438464-20230717.png)」のように、結果のスクリーンショットを効果的に使用。
- **ツイート埋め込み**: 関連する議論や意見を示すためのツイート。「@[tweet](https://twitter.com/hagakun_yakuzai/status/1495710538105606145)」のように、自身や他者のツイートを埋め込む。
- **mermaid図**: フローチャートや関係図を視覚化。「```mermaid」を使用して図表を作成。「graph TD」などでグラフ構造を表現。

### 特殊な要素
- **メッセージブロック**: :::message で囲まれた補足情報。注意点や追加情報を提供する際に使用。「:::message マーケットプレイスで探すといくつか出てくるので注意。Alexander氏の作成した拡張の方が使いやすい。:::」
- **詳細ブロック**: :::details で折りたたまれた詳細情報。長いコードブロックや詳細な実装例を隠す際に使用。「:::details　Schemaの詳細」「:::details Goのdataloaderの実装」
- **mermaid図**: フローチャートや関係図の視覚化。システム構成やデータフローを説明する際に使用。「```mermaid graph TD Flutter --> GraphQL GraphQL --> NestJS GraphQL --> Go NestJS --> Docker Go --> Docker Docker --> MySQL ```」
- **引用ブロック**: 「>」を使った引用ブロック。他の情報源からの引用や、特に強調したい内容を示す際に使用。「> あなたが何歳で、どのような人生をおくってきたにせよ、やりたい創作がどんなことだとしても、自分の創造性をひきだすのに遅すぎることはない。」

## 文章構造とパターン

### 典型的な文章構造
- **導入部**: 問いかけや現状認識から始まることが多い。「レビューは得意ですか？」「LLMは高度な知的タスクを実行可能である。そんな理解が世界に広まっていく中で...」
- **本論部**: 見出しを多用し、階層的に情報を整理。各セクションは結論から始まり、その後に詳細や根拠を展開する構造。
- **結論部**: 実践的な提案や今後の展望で締めくくることが多い。「ぜひ一番最初に検討することをオススメします」「ぜひとも使ってみて、良いチューニングなどがあれば共有お願いします！」

### 特徴的な文章パターン
- **問題提起→解決策提示**: 「シフトをぽちぽちと手打ちで作るのはあまりにもムダが多すぎます。」→「ということで作ってみました！」
- **自問自答**: 「なぜ、そんなことをするのか。それは、アウトプットが楽しいから」
- **対比構造**: 「勉強はクソだ。やりたくないものは得られるものが少ない。でも、学びはすごいぞ」
- **譲歩→主張**: 「確かに、メルカリのような大きな組織...では、おそらく推奨の通りにするのが正しいでしょう。一方で、小さな組織、限られたエンジニアリソースを活かそうとすると...」
- **具体例→一般化**: 具体的な実装例や体験を示した後、一般的な原則や教訓を導き出す。

### 段落構成の特徴
- **短い段落**: 1〜3文程度の短い段落が多く、読みやすさを重視。
- **段落間の接続**: 「さて」「ところで」「一方」などの接続語を使って、段落間の関係を明確にする。
- **視点の切り替え**: 技術的な説明と個人的な体験を交互に配置し、リズム感を生み出す。

### 文末表現の特徴
- **断定**: 「〜です」「〜だ」などの断定的な文末。
- **推量**: 「〜でしょう」「〜かもしれません」など、読者への配慮を示す表現。
- **感情表現**: 「〜ですね！」「〜なんです」など、感情を込めた文末。
- **問いかけ**: 「〜ではないでしょうか？」「〜と思いませんか？」など、読者に考えを促す文末。
- **余韻**: 「〜」で終わる文で余韻を残す。

### 文と文のつなぎ方
- **接続詞の活用**: 「しかし」「また」「さらに」「したがって」などを効果的に使用。
- **列挙**: 「まず」「次に」「最後に」などで順序立てて説明。
- **因果関係**: 「〜なので」「〜だから」「〜ため」などで因果関係を明確に。
- **対比**: 「〜一方で」「〜に対して」などで対比を示す。
